# Ian 小黑怪誕內文配圖

## 核心定位

為中文文章設計和生成 16:9 橫式內文配圖。目標不是做商業插畫、PPT 資訊圖或可愛卡通，而是把文章裡的關鍵判斷、流程、結構、狀態或隱喻，變成一張清爽、怪誕、有創意、可讀但不說明書的手繪解釋圖。

預設視覺 IP 是「小黑」：黑色實心、白點眼、細腿、空表情，認真做一件荒誕但成立的事。小黑必須參與畫面的核心動作，不能只是站在旁邊當裝飾。

## 先讀這些參考

按任務需要讀取，不要一次塞滿上下文：

- `references/style-dna.md`：風格 DNA、顏色、文字、禁忌。
- `references/xiaohei-ip.md`：小黑 IP 的形象、性格、動作庫和禁忌。
- `references/composition-patterns.md`：結構類型、原創隱喻方法和避免複刻規則。
- `references/prompt-template.md`：單張圖片生成提示詞模板。
- `references/qa-checklist.md`：生成後檢查和迭代規則。
- `assets/examples/`：只作少量視覺校準，不進入預設生成路徑。不要照抄這些案例的構圖、物件或標註。

## 工作流程

### 1. 消化內文

先讀使用者給的內文、連結、Notion 頁面、Markdown 檔案或截圖內容。提煉：

- 核心觀點是什麼
- 哪些段落承擔認知轉折
- 哪些內容適合用圖解釋
- 哪些地方只適合文字，不需要圖

不要平均配圖。優先選擇「認知錨點」，例如：核心判斷、兩個斷點、輸入輸出閉環、分流、前後對比、一魚多吃、承接路徑、常見坑、角色狀態變化。

### 2. 先出配圖策略

如果使用者只是說「分析怎麼配圖 / 思考哪些地方需要配圖」，先給 shot list。每張圖寫清楚：

- 放在哪個段落後
- 圖的主題
- 核心意思
- 結構類型
- 小黑在圖裡做什麼
- 建議元素
- 建議中文標註詞

預設 4-8 張。文章很短時 1-3 張；長文也不要輕易超過 9 張。夠用就好，避免把內文做成畫冊。

### 3. 單張生成

如果使用者明確要求「生成 / 輸出 / 做圖 / 幫我生成」，不要停下來等確認；用內建 `image_gen` 每張單獨生成。不要把多張圖拼在一張裡。

每張圖只講一個核心結構。提示詞必須包含：

- 16:9 橫式中文內文配圖
- 純白背景
- 黑色手繪線稿
- 少量紅色/橙色/藍色中文手寫批註
- 大量留白
- 小黑作為核心動作主體
- 禁止 PPT、商業插畫、幼稚可愛、複雜架構、左上角類型標題

不要復刻過往案例。案例只提供風格密度和小黑參與方式，不能直接重複使用「傳送帶斷點 / 小黑拉線 / 素材魚 / 蓋章工具箱 / 常見坑路徑」等已有構圖，除非使用者明確要求復刻某張圖。每次都要從當前文章重新發明一個奇怪但成立的隱喻。

### 4. 檢查與迭代

生成後檢查 `references/qa-checklist.md`。如果出現以下問題，優先重生成或局部編輯：

- 小黑只是裝飾
- 畫面太滿
- 太像流程圖/PPT
- 中文太多或錯字嚴重
- 左上角出現「常見坑/流程圖/系統架構圖」等標題
- 畫風太可愛、幼稚、死板
- 背景不是乾淨白底

### 5. 儲存交付

如果使用者在 workspace 內工作，把最終圖複製到：

```text
assets/<article-slug>-illustrations/
```

按順序命名：

```text
01-topic-name.png
02-topic-name.png
```

保留原始生成檔案，不要覆蓋已有資產，除非使用者明確要求替換。

## 輸出規範

所有中文輸出一律使用台灣慣用繁體中文，包含 shot list、圖片標註建議、交付說明、檔名說明與改圖指示。避免簡體中文、陸用詞與簡體標點進入使用者可見內容。

生成前的策略輸出要簡短精準。生成後的交付要包含：

- 生成了幾張
- 每張圖的用途
- 儲存路徑
- 哪些圖最穩定，哪些圖是可選

不要長篇解釋風格理論；讓圖自己說話。
